docker容器卷

建立容器卷（以命令的方式）

1、docker run -it -v /myDataVolume:/dataVolumeContainer + 镜像名

​ myDataVolume是宿主机上根目录下创建的，dataVolumeContainer是容器内部根目录下创建的，它们彼此之间能够共享数据，读写数据都可以，即使容器关闭，数据也会共享，这就是持久化，全量复制

2、docker run -it -v /myDataVolume:/dataVolumeContainer:ro + 镜像名

​ ro:代表这个是只读的，也就是在容器内部的dataVolumeContainer这个文件只能读取宿主机修改的内容，而不能自己修改，也就是写操作只能单向

建立容器卷（以dockerfile的形式）

容器卷之间的继承

1、 首先运行一个名叫dc01的容器，docker run -it –name dc01 zyh/centos

2、 因为我们自己创建的镜像里面已经默认有了两个容器卷，所以新创建的容器dc01中也会默认有那俩个容器卷

3、 以dataVolumeContainer2为例，先进入文件中，创建个名为dc01_add.txt的文件

4、 然后再运行docker run -it –name dc02 –volumes-from dc01 zyh/centos 来进行继承

5、 运行之后dc02中也有这两个容器卷，并且dataVolumeContainer2中也有dc01_add.txt文件

6、 然后在dc02中建立个dc02_add.txt，然后dc01里面的dataVolumeContainer2中也会有这个dc02_add.txt文件

7、 再docker run -it –name dc03 –volumes-from dc01 zyh/centos 创建dc03的容器，之后在dc03中的dataVolumeContainer2中建立个dc03_add.txt文件

8、 之后三个容器中都会有这三个一模一样的共享文件，这就实现了共享

注意：如果此时删除了dc01这个父容器（前提是：dc02跟dc03继承的都是dc01，dc03继承的并不是dc02），那dc02跟dc03中的文件还会存在，进一步，如果在dc02中再创建一个文件，那dc03中也会出现这个文件，这就是全方位的共享。

dockerfile的体系结构

1、 FROM：基础镜像，表示当前镜像是基于哪个父镜像

2、 MAINTAINER：镜像维护者的姓名与邮箱地址

3、 RUN：容器构建的时候需要执行的额外命令

4、 EXPOSE：当前容器对外暴露的端口号

5、 WORKDIR：表示容器启动之后工作的目录，也就是落脚点，不指定的话就代表根目录

6、 ENV：用来在构建镜像过程中设置环境变量

7、 ADD：就是从宿主机上拷贝一个文件到镜像，可以是压缩包，它会自行解压

8、 COPY：跟ADD作用一样，但是不会自行解压，只会拷贝

9、 VOLUME：建立容器数据卷，用于数据保存和持久化

10、CMD：指定一个容器启动时运行的命令，Dockerfile中可以有多个CMD命令，但只有最后一个命令生效，最后一个CMD命令会被docker run之后的参数替换掉

11、ENTRYPOINT：跟CMD作用一样，只不过docker run之后的参数不会被替换，而是会被追加到命令之中执行

12、ONBUILD：当构建一个被继承的Dockerfile时运行命令，父镜像在被子镜像继承后，父镜像的onbuild就会被触发

13、USER：设定哪个用户可以运行该dockerfile文件

14、.dockerignore：忽略一些东西

redis事务

1、本质：可以一次执行多条命令，本质是一组命令。一个事务中的所有命令都会序列化，按顺序的串行化执行，而不会被其它命令插入，不许加塞

2、特点：不保证原子性，部分支持事务

3、开启事务：multi（可以一次性执行多条命令），开启事务之后，对数据库进行的操作都会返回一个queued，代表已经入队列，但没有执行，再输入exec，就会依次执行。

4、放弃事务：discard，一旦输入，就会放弃之前的批操作

5、全体连坐：一旦在批操作中有一个因为语法出错（还未exec提交，批操作中就已经提示的错误），该批次操作会全部作废

6、冤头债主：一旦在批操作中有一个因为字符类型设定出错（比如set k1 aa，incr k1，k1的value值是字符型，不能加1），虽然这句话有错误，但是当时exec提交之前并不会直接报错，exec之后就只有这一个操作不会执行，其他的都会执行

7、watch监控：监控应先在开启事务multi之前执行，如果在提交事务exec之前有人加塞，执行就会失败

8、unwatch：取消watch对所有key的监控，在发现自己数据被改动之后，先unwatch，然后再watch，继续执行自己的事务操作

9、悲观锁：顾名思义，每次操作数据的时候都认为别人会修改，所以每次修改数据的时候就上锁，这样别人就修改不了了，虽然保证了数据的移植性，但是效率太低，比如行锁、表锁、读锁、写锁等

10、乐观锁：顾名思义，每次操作数据的时候都认为别人不会修改，所以不会上锁，但是会加上版本号version的机制，策略：提交的版本必须 > 记录中的当前版本才会执行更改数据操作

缓存穿透和雪崩

缓存穿透（查询不到）

概念：

​ 用户想要查询一个数据，发现Redis内存数据库没有，也就是缓存没有命中，于是向持久层数据库查询，发现也没有，就导致查询失败。当用户很多的时候，缓存都没有命中（秒杀），于是都去请求持久层数据库，会给数据库曹成很大的压力，这时候就会出现缓存穿透。

解决方法：

​ 1、加入一个布隆过滤器：进行校验，不符合则丢弃

​ 2、设置一些缓存空对象：当存储层不命中时，即使返回的对象为空也将其缓存起来，同时设置过期时间，之后再访问就会从缓存中取，保护了后端数据

缓存击穿（查到的太多了）

概念：

​ 是指一个很热的key点，在不停的扛着大并发，大并发针对这个点进行访问，当这个key值失效的的瞬间，持续的大并发就会击穿缓存，直接请求数据库，就像一个大屏幕凿开了一个洞。

​ 当某个key在过期的瞬间，有大量请求并发访问，直接请求数据库查询新数据，并且回写缓存，会导致数据库压力瞬间加大

解决方法：

​ 1、设置热点永不过期：不是很好

​ 2、加分布式锁：使用分布式锁，保证一个key同时只能只有一个线程来访问，其它线程没有获得分布式锁的权限，因此需要等待即可。这种方式将压力转移到了分布式锁，因此对分布式锁的考验很大。

缓存雪崩

概念：

​ 指某一时间段内，缓存集体过期失效，导致Redis宕机，有可能是自己设置的过期时间不合适，也有可能是缓存服务器某个节点宕机或者断网，这是最可怕的。

解决方法：

​ Redis高可用：多设置几台Redis缓存服务器

​ 停止其它去关紧要服务：比如双十一，退款服务就会停掉，来解决压力

​ 限流降级：在缓存失效以后，通过加锁或者队列来控制数据库写缓存的线程数量，利用hystrix来实现限流降级

​ 数据预热：在正式部署之前，把可能的数据先预先访问一遍，这样部分可能大量访问的数据事先加载到缓存中。在将来要发生大并发访问前手动触发加载不同的key，设置不同的过期时间，让缓存失效时间尽量均匀点
